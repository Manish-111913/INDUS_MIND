# IndusMind — Production Runbook

AWS deployment operations for the backend (docs/02 §48 layout: ECS Fargate + RDS
Postgres 16 + ElastiCache + S3 + Secrets Manager, 2-AZ VPC). The Terraform in
[`infra/terraform`](./terraform) provisions everything below.

> **Status:** this is a *credible skeleton*. The HCL is `terraform validate`-clean
> and models the real §48 topology, but has not been `apply`-ed to a live account.
> `TODO(prod)` markers flag the account-specific values (image tags, ACM cert,
> domain) you must supply before a real apply.

---

## 0. Prerequisites

- Terraform ≥ 1.6, AWS CLI v2, Docker.
- An AWS account with an IAM principal allowed to create VPC/ECS/RDS/ElastiCache/
  S3/Secrets/IAM resources.
- A remote-state backend: an S3 bucket + DynamoDB lock table (create once, pass via
  `-backend-config`).
- An ACM certificate for `api.<your-domain>` in the deployment region.

## 1. Provision infrastructure

```bash
cd infra/terraform
cp terraform.tfvars.example terraform.tfvars    # fill in image tags, ACM ARN, domain

terraform init \
  -backend-config="bucket=<state-bucket>" \
  -backend-config="key=indusmind/prod.tfstate" \
  -backend-config="region=<region>" \
  -backend-config="dynamodb_table=<lock-table>"

terraform plan -out tf.plan
terraform apply tf.plan
```

Key outputs (`terraform output`): `alb_dns_name`, `rds_endpoint`,
`redis_endpoint`, `ecr_api_repository_url`, `ecr_worker_repository_url`,
`ecs_cluster_name`, `app_secret_arn`, `alerts_topic_arn`.

### 1a. Write real secrets

Terraform seeds `${project}/app` with placeholders (and `ignore_changes` so it
won't clobber real values). After apply, write the real bundle **once**:

```bash
aws secretsmanager put-secret-value --secret-id "$(terraform output -raw app_secret_arn)" \
  --secret-string '{"SECRET_KEY":"...","JWT_PRIVATE_KEY":"...","JWT_PUBLIC_KEY":"...","ANTHROPIC_API_KEY":"...","OAUTH_GOOGLE_CLIENT_ID":"...","OAUTH_GOOGLE_CLIENT_SECRET":"..."}'
```

`DATABASE_URL` is generated by Terraform from the RDS endpoint + generated
password — no manual step. Subscribe an email/Slack endpoint to
`alerts_topic_arn` for CloudWatch alarms.

## 2. Build & push images

`.github/workflows/backend.yml` builds and tests on every push; the CD extension
tags images with the git SHA and pushes to ECR:

```bash
ACCOUNT=<id>; REGION=<region>; SHA=$(git rev-parse --short HEAD)
aws ecr get-login-password --region $REGION | docker login --username AWS --password-stdin $ACCOUNT.dkr.ecr.$REGION.amazonaws.com

docker build -t $ACCOUNT.dkr.ecr.$REGION.amazonaws.com/indusmind-api:$SHA    -f BACKEND/Dockerfile        BACKEND
docker build -t $ACCOUNT.dkr.ecr.$REGION.amazonaws.com/indusmind-worker:$SHA -f BACKEND/Dockerfile.worker BACKEND
docker push $ACCOUNT.dkr.ecr.$REGION.amazonaws.com/indusmind-api:$SHA
docker push $ACCOUNT.dkr.ecr.$REGION.amazonaws.com/indusmind-worker:$SHA
```

Set the pushed refs as `api_image` / `worker_image` in `terraform.tfvars`.

## 3. Database migration (one-off task before rollout)

Migrations run as a **one-off ECS task** using the API image, *before* the new
service revision is deployed (docs/02 §56). This creates the `pgvector` /
`pg_trgm` / `pgcrypto` extensions and applies the schema.

```bash
CLUSTER=$(terraform output -raw ecs_cluster_name)
aws ecs run-task --cluster "$CLUSTER" \
  --launch-type FARGATE \
  --task-definition indusmind-prod-api \
  --network-configuration "awsvpcConfiguration={subnets=[<private-subnets>],securityGroups=[<app-sg>]}" \
  --overrides '{"containerOverrides":[{"name":"api","command":["alembic","upgrade","head"]}]}'
```

Wait for the task to reach `STOPPED` with exit code 0 before continuing.
(Optionally seed reference data once: override command with `python -m seeds.seed`.)

## 4. Deploy application

Force a new deployment so ECS pulls the new task revision. The service has a
**deployment circuit breaker with rollback** (§56), so a failed rollout reverts
automatically.

```bash
aws ecs update-service --cluster "$CLUSTER" --service indusmind-prod-api    --force-new-deployment
aws ecs update-service --cluster "$CLUSTER" --service indusmind-prod-worker --force-new-deployment
aws ecs update-service --cluster "$CLUSTER" --service indusmind-prod-beat   --force-new-deployment
```

### Smoke test

```bash
curl -fsS https://api.<domain>/healthz          # {"status":"ok"}
curl -fsS https://api.<domain>/readyz           # 200 when pg+redis+neo4j+s3 healthy
curl -fsS https://api.<domain>/metrics | head   # Prometheus series
```

Point the `api.<domain>` Route53 alias at `alb_dns_name`.

## 5. Environment variable mapping (local → AWS)

Everything switches by env var only — the `core/` adapter pattern guarantees it
(docs/00 §7, docs/02 §49). Secrets come from Secrets Manager; the rest are set as
plain task environment by Terraform.

| Variable | Local (`.env`) | Production | Source |
|---|---|---|---|
| `APP_ENV` | `local` | `prod` | task env |
| `DATABASE_URL` | `…@localhost:5433/indusmind` | RDS Postgres 16 Multi-AZ endpoint | **Secrets Manager** (`…/database-url`) |
| `REDIS_URL` | `redis://localhost:6380/0` | ElastiCache primary endpoint | task env (from TF output) |
| `NEO4J_URI` / `USER` / `PASSWORD` | Neo4j container | **Neo4j Aura** | Secrets Manager |
| `S3_ENDPOINT` / `S3_BUCKET` | MinIO `http://localhost:9000` | S3 (native endpoint) + CloudFront | task env |
| `S3_ACCESS_KEY` / `SECRET_KEY` | `minioadmin` | *(unset — task role IAM)* | IAM task role |
| `LLM_PROVIDER` / `ANTHROPIC_API_KEY` | key or Ollama | API key | Secrets Manager |
| `OCR_PROVIDER` | `paddle` | `textract` | task env |
| `SECRET_KEY`, `JWT_PRIVATE_KEY`, `JWT_PUBLIC_KEY` | dev defaults | strong secrets | Secrets Manager (`…/app`) |
| `OAUTH_GOOGLE_CLIENT_ID` / `SECRET` | dev creds | prod OAuth app | Secrets Manager |
| `CORS_ORIGINS` | `http://localhost:3000` | `https://app.<domain>` | task env (`var.cors_origins`) |
| `SENTRY_DSN` | empty (off) | project DSN | Secrets Manager |
| `RATE_LIMIT_ENABLED` | `true` (off in tests) | `true` | task env |

RDS at rest, S3, ElastiCache and EBS are KMS-encrypted; ALB terminates TLS 1.2+
via ACM (docs/02 §42).

## 6. Disaster recovery (docs/02 §53)

**Targets:** RPO ≤ 15 min (RDS PITR), RTO ≤ 4 h (single-region), ≤ 24 h (region loss).

**Backups (§52):** RDS automated snapshots + PITR (14-day window here) & weekly
logical dump to S3; S3 versioning + replication; Neo4j nightly dump **and**
rebuild-from-Postgres (the graph is a projection); Redis is not backed up (cache).

**Single-region recovery runbook:**
1. Restore RDS from the latest snapshot or PITR to the target timestamp
   (`aws rds restore-db-instance-to-point-in-time`).
2. Repoint `…/database-url` in Secrets Manager at the restored endpoint.
3. Redeploy ECS services — images are immutable in ECR, so the exact tested build
   redeploys (`aws ecs update-service --force-new-deployment`).
4. Rebuild the knowledge graph from Postgres (`python -m seeds.rebuild_graph` as a
   one-off task) — no separate Neo4j restore needed.
5. S3 is already replicated; no action.
6. Run the smoke suite (§4) and verify `/readyz` is green.

**Region loss:** restore RDS + S3 from cross-region-replicated backups into the
secondary region, `terraform apply` the stack there (state/secrets replicated),
then follow steps 2–6. Drill this once and screenshot for the deck.
